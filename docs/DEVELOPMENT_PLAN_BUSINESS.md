## 二、P0 级别：核心保命功能（建议 1 个月内补齐）

### P0-1. 人员与组织：项目经理绑定与数据隔离

**业务痛点：**  
工程行业项目利润高度敏感。若项目经理仅为文本字段，全员可见全公司项目底牌，推行阻力极大。

**改进要求：**

- 将 `Project.managerId` 绑定系统用户（User 表）：存储用户 id 或 username，与用户管理一致。
- 实现**数据隔离**：项目经理登录后只能查看/操作**自己负责的项目**；高管/财务保留全局视图。
- 项目列表与项目表单中「项目经理」改为**用户下拉选择**，禁止手输文本。

**可交付项（供 Plan Agent 拆任务）：**

- 数据模型：`managerId` 改为关联 User（或存 userId/username，与现有角色权限一致）。
- 后端：项目列表/详情接口按当前用户角色过滤（pm 仅能看 managerId = 当前用户的项目）。
- 前端：项目表单「项目经理」为可搜索用户下拉；项目列表对 pm 仅展示其负责项目。

---

### P0-2. 财务与报表规范：基础数据标准化

**业务痛点：**  
支出类别若为自由文本，月底报表无法统计（如“买砖” vs “材料采购”），各项成本占比失真。

**改进要求：**

- 支出类别改为**下拉选择**，禁止自由文本。
- 建议枚举（可配置）：**人工费、材料费、机械费、分包费、间接管理费**等；后台写死与「项目成本类型」的映射（material/labor/other），杜绝垃圾数据。
- 前端录入支出时只能选枚举值；报表与项目成本回写均基于该枚举。

**可交付项（供 Plan Agent 拆任务）：**

- 后端：支出类别枚举或配置表；`FinanceRecord.category` 约束为枚举值；类别 → costType 映射写死在服务层（或配置表）。
- 前端：财务收支表单「类别」为下拉；若现有为输入框则改为下拉并保留与 costType 的映射。
- 文档：在用户手册或配置中明确「支出类别与项目成本类型对应表」。

---

### P0-3. 合同与回款：应收账款（催款）管理

**业务痛点：**  
工程公司死于缺钱。只记“已收款”是事后统计，老板需要知道“明天该去催哪笔钱”。

**改进要求：**

- 增加**回款计划**与**到期提醒**：
  - 录入/编辑项目时，可拆分**收款节点**（如预付款、进度款、结算款、质保金），每节点含：计划收款日、计划金额、状态（待收/部分收/已收）。
  - 与现有 `contractAmount`、`receivedAmount` 及财务收入记录可关联（如回款计划某节点“已收”后，可关联到具体财务收入单或自动汇总到已收款）。
- 系统首页（或仪表盘）必须有**【近期待催款预警】**：如未来 7/15 日内计划收款未收齐的项目与金额列表。

**可交付项（供 Plan Agent 拆任务）：**

- 数据模型：新增「回款计划」实体（如 PaymentPlanItem），字段含：项目 id、节点名称、计划日期、计划金额、已收金额、状态等；与 Project 关联。
- 后端：回款计划 CRUD；按“计划日临近且未收齐”查询，供首页预警。
- 前端：项目表单中“回款计划”子表（增删改节点）；仪表盘/首页「近期待催款预警」区块。
- 可选：回款计划节点与财务收入单关联（如勾选某笔收入计入某节点），便于自动更新“已收金额”。

---

### P0-4. 成本口径统一（出库 vs 财务支出）

**业务痛点：**  
仓库出料算不算成本？财务付给供应商的钱算不算？口径不统一会导致项目经理与财务长期争执。

**改进要求：**

- **开会定死规则**。建议口径：**项目材料成本 = 财务直发项目的采购支出 + 仓库领用出库的材料折算金额**（两部分均参与汇总）。
- 项目详情页**清晰展示两部分数据来源**：
  - 来自财务支出（按 projectId + 成本类型 material 的已审批支出汇总）；
  - 来自出库（按 projectId 的已审批出库 qty×price 汇总）；
  - 总材料成本 = 二者之和（或按既定规则展示），做到账实相符。

**可交付项（供 Plan Agent 拆任务）：**

- 后端：项目详情接口返回「财务材料成本」「出库材料金额」及「材料成本合计」；若当前 materialCost 仅来自财务，则出库部分单独字段或计算字段，不破坏现有回写逻辑。
- 前端：项目详情「项目成本分析」或「成本构成」中分项展示：财务支出材料成本、出库材料金额、合计；并注明数据来源。
- 产品/业务文档：明确「项目成本口径说明」（材料 = 财务 + 出库；人工/其他仅财务），写入用户手册或开发计划附录。

**技术现状参考：**  
当前已实现：财务支出审批通过后按 costType 回写 `Project.materialCost/laborCost/otherCost`；收入审批通过回写 `receivedAmount`。出库仅按项目汇总金额在详情页展示，未写入 materialCost。P0-4 要求在展示层统一口径并分来源展示，是否回写出库到 materialCost 由既定规则决定（建议与业务确认后写入文档）。

## 三、P1 级别：管理进阶功能（建议 1–3 个月内逐步完善）

### P1-1. 供应商 / 分包商：应付账款管理

**业务痛点：**  
“两头拿捏”：催甲方回款、拖乙方付款。无供应商管理则无法回答“我们还欠张老板多少钱”。

**改进要求：**

- 建立**客商档案**（分包商、材料商）：名称、联系人、银行信息等主数据。
- 采购入库、财务支出**必须关联到具体供应商**（下拉选择）。
- 生成**【供应商应付/已付/欠款明细表】**，供老板签字付款前查阅。

**可交付项（供 Plan Agent 拆任务）：**

- 数据模型：新增 Supplier（或 Vendor）实体；StockLog（入库）、FinanceRecord（支出）增加 supplierId 或等效关联。
- 后端：供应商 CRUD；按供应商汇总应付/已付/欠款接口；报表接口。
- 前端：供应商管理页；入库/支出表单选择供应商；应付/已付/欠款明细报表页。

---

### P1-2. 预算与成本管控：防漏底的红线

**业务痛点：**  
没有预算的管控是“记账”，有预算才是“管理”。项目经理缺少参照易超支。

**改进要求：**

- 建项目时强制或强烈建议录入**控制预算**（总预算或分项：材料/人工/其他）。
- 当实际成本（财务支出 + 出库，与 P0-4 口径一致）达到预算的 **80%** 时**黄灯预警**，**100%** 时**锁定或需特批**（如不允许再关联该项目的支出/出库，或需管理员放行）。

**可交付项（供 Plan Agent 拆任务）：**

- 数据模型：Project 增加预算字段（如 totalBudget 或 materialBudget/laborBudget/otherBudget）。
- 后端：在创建/更新支出、出库时校验项目“是否已超预算/是否锁定”；达 80%/100% 时写日志或状态，供预警与权限判断。
- 前端：项目表单预算录入；项目详情/仪表盘超支预警（黄/红）；超 100% 时拦截或提示需特批。

---

### P1-3. 工程管理扩展：变更与签证管理

**业务痛点：**  
“中标靠低价，盈利靠签证”。工程量增加若不及时记录，结算时易吃亏。

**改进要求：**

- 增加**【变更/签证单】**模块：签证事由、预估增加金额、甲方确认状态等；审批通过后可**自动追加到合同金额或项目总收入预测**（与现有 `contractAmount` 或扩展“预估总收款”一致）。

**可交付项（供 Plan Agent 拆任务）：**

- 数据模型：新增 ChangeOrder 或 Visa 实体，关联 Project，含金额、状态、审批信息。
- 后端：变更单 CRUD 与审批；审批通过后更新项目“合同金额”或“预估总收”（业务规则二选一或并存）。
- 前端：变更/签证单列表与表单；与项目详情联动展示。

---

## 四、P2 级别：锦上添花（业务稳定后再考虑）

### P2-1. 工程文件/图纸管理

**评估：**  
图纸与资料体量大，初期上传系统会带来存储与性能压力。

**建议：** 暂缓。用钉钉文档/企业微信微盘等建立项目文件夹替代，系统内仅保留“项目文档清单”或链接即可。

---

### P2-2. 复杂项目看板 / 甘特图

**评估：**  
一线项目经理极少天天拖拽甘特图；现有「里程碑」已能满足大部分进度把控。

**建议：** 暂缓。优先做**里程碑超期预警**（计划日期已过但未完成），比甘特图更实用。

---

### P2-3. 撤销审批与红字冲销

**评估：**  
财务/库存单据生效后，简单“一键撤销”会导致数据错乱（如材料已领用但库存回滚）。

**建议：**  
不做简单撤销。引入**红字冲销**逻辑：发现错误时，新做一张**负数/冲销单据**抵消原单，保留全部操作痕迹，符合财务规范。若实现，需在需求中明确冲销规则（谁可冲销、是否需审批、与原单的关联方式）。

---

## 五、落地建议（给一线与管理层）

- **先跑通核心账：** 不追求大而全，先把 P0 打通。让一线只做三件事：录合同收款节点、走收支审批、做物资出入库。
- **定死口径：** 召开项目经理、财务、库管碰头会，按本文档 P0-4 与现有逻辑，当场敲定「项目成本按哪个算」并写入配置或用户手册。
- **老板驱动：** 规定“不在系统里的回款计划不发提成，不在系统里的欠款不签字付款”，便于推行。

## 七、技术现状简要对照（便于开发与 Plan Agent 对齐）

- **项目成本与已收款：** 已实现财务审批通过后按 costType 回写 `Project.materialCost/laborCost/otherCost`，收入回写 `receivedAmount`；`FinanceRecord` 已有 `costType`；支出类别枚举与 `GET /api/finance/categories` 已实现（P0-2）；项目详情返回 `materialCostFromFinance`/`materialCostFromStock`/`materialCostTotal` 并分项展示（P0-4）。
- **项目经理：** `Project.managerId` 存 username，与 User 关联；项目列表/详情按 pm 过滤；`GET /api/users/options`；项目表单项目经理为用户下拉（P0-1 已完成）。
- **回款计划：** 已新增实体 `PaymentPlanItem`、CRUD `/api/payment-plans`、`GET /api/payment-plans/upcoming`；项目详情回款计划子表与仪表盘近期待催款预警已实现（P0-3 已完成）。**回款计划节点与收入关联**：收入支持 `paymentPlanItemId`，回款计划节点的已收金额按关联的已审批收入自动汇总。
- **供应商：** 已新增实体 `Supplier`、CRUD `/api/suppliers`、入库/支出可选 `supplierId`；`GET /api/suppliers/{id}/balance` 与 `GET /api/suppliers/reports/balance` 应付/已付/欠款汇总（P1-1 已完成）。
- **预算：** `Project` 已新增 `totalBudget`；项目详情/列表返回 `actualCostTotal`、`budgetRatio`、`budgetAlertStatus`（80% 黄灯、100% 红灯）；支出/出库超 100% 时校验，仅管理员可特批（P1-2 已完成）。
- **变更/签证单：** 已新增实体 `ChangeOrder`、CRUD `/api/change-orders`、`POST /api/change-orders/{id}/approve`；审批通过后回写 `Project.contractAmount`；项目详情可查看关联变更单（P1-3 已完成）。
- **里程碑超期预警：** 已新增接口 `GET /api/milestones/overdue`，返回当前用户可见项目下计划日期已过且未完成的里程碑；前端仪表盘展示「里程碑超期预警」区块，项目详情里程碑列表对超期节点标红显示（P2-2 建议已落地为轻量版预警）。
- **红字冲销：** 财务冲销与库存冲销均已实现。财务冲销：`FinanceRecord.reversalOfId`、`isReversal`；创建冲销单走审批流程（status=pending），审批通过后按负金额回减项目成本/已收款；仅财务/管理员角色可发起。库存冲销：`StockLog.reversalOfId`、`isReversal`；`POST /api/stock/reversal` 创建冲销单，`POST /api/stock/{id}/approve-reversal` 审批；出库冲销回增库存，入库冲销回减库存；仅管理员/库管可发起，管理员审批。

以上可作为产品与开发排期、任务拆分的直接依据；Plan Agent 可据此输出迭代计划与 Story/任务列表。

## 八、验收标准（按优先级）

以下为各功能上线前需满足的验收条件，可用于迭代评审与测试用例设计。

### P0 验收标准

| 编号   | 验收项           | 通过条件                                                                                                                                      |
| ------ | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| P0-1.1 | 项目经理绑定用户 | 项目保存后 managerId 对应系统用户 id 或 username；项目列表/详情接口返回的经理信息与用户表一致。                                               |
| P0-1.2 | 数据隔离         | 以 pm 角色登录时，项目列表仅包含 managerId = 当前用户的项目；admin/finance 可见全部项目。                                                     |
| P0-1.3 | 前端选人         | 项目表单「项目经理」为下拉（可搜索），仅能选系统用户，无法手输任意文本。                                                                      |
| P0-2.1 | 类别枚举         | 财务支出「类别」仅为预定义枚举值（如人工费、材料费、机械费、分包费、间接管理费）；后端拒绝非法 category。                                     |
| P0-2.2 | 成本类型映射     | 每条支出对应明确 costType（material/labor/other）；报表与项目成本回写按该映射统计无误。                                                       |
| P0-2.3 | 前端下拉         | 财务收支录入界面类别为下拉选择，无自由文本输入。                                                                                              |
| P0-3.1 | 回款计划 CRUD    | 项目可维护多条回款计划节点（节点名称、计划日期、计划金额、已收金额、状态）；支持增删改。                                                      |
| P0-3.2 | 待催款预警       | 仪表盘/首页有「近期待催款」区块，展示未来 N 日（如 7/15 日）内计划收款未收齐的项目与金额；数据与回款计划一致。                                |
| P0-3.3 | 与已收款关联     | 回款计划节点状态或已收金额可与财务收入或项目已收款联动（具体规则以产品为准）。                                                                |
| P0-4.1 | 项目详情成本分项 | 项目详情页展示：来自财务的材料成本、来自出库的材料金额、材料成本合计；并注明数据来源。                                                        |
| P0-4.2 | 口径一致         | 财务材料成本 = 该项目已审批且 costType=material 的支出汇总；出库材料金额 = 该项目已审批出库的 qty×price 汇总；合计 = 二者之和（或已定规则）。 |

### P1 验收标准

| 编号   | 验收项           | 通过条件                                                                            |
| ------ | ---------------- | ----------------------------------------------------------------------------------- |
| P1-1.1 | 客商档案         | 供应商主数据可维护（名称、联系人、银行信息等）；入库与支出可关联供应商。            |
| P1-1.2 | 应付/已付/欠款表 | 存在报表或接口：按供应商汇总应付、已付、欠款；数据与入库/支出关联一致。             |
| P1-2.1 | 预算录入         | 项目可录入控制预算（总预算或分项）；项目详情/列表可展示预算与实际对比。             |
| P1-2.2 | 80% 预警         | 当实际成本达预算 80% 时，有黄灯或明确预警展示（项目详情/仪表盘/消息）。             |
| P1-2.3 | 100% 锁定或特批  | 当实际成本达预算 100% 时，新支出/出库关联该项目时被拦截或需管理员特批；规则可配置。 |
| P1-3.1 | 变更/签证单      | 变更单可创建、编辑、审批；含事由、金额、状态、关联项目。                            |
| P1-3.2 | 审批后更新项目   | 变更单审批通过后，项目合同金额或预估总收按约定规则更新；项目详情可查看关联变更单。  |

## 九、接口清单（API/后端）

以下为建议新增或扩展的后端接口，供 Plan Agent 与后端开发对齐；具体路径与 REST 风格以项目现有规范为准。

### P0 相关

| 功能 | 接口/能力                                                                                 | 说明                                                                                                                  |
| ---- | ----------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| P0-1 | `GET /api/projects`                                                                       | 支持按当前用户过滤：若角色为 pm，仅返回 managerId = 当前用户的项目；需传递或从 Token 解析当前用户。                   |
| P0-1 | `GET /api/users` 或 `/api/users/options`                                                  | 供前端「项目经理」下拉使用；可返回 id、username、角色等，需满足权限。                                                 |
| P0-1 | `Project.managerId` 类型/含义                                                             | 若改为 userId，需在项目创建/更新时校验 userId 存在且可选；列表/详情返回经理姓名时需关联查询 User。                    |
| P0-2 | 支出类别枚举                                                                              | `GET /api/finance/categories` 或配置接口，返回允许的类别列表及与 costType 的映射；或写死在前后端约定文档。            |
| P0-2 | `POST/PUT /api/finance`                                                                   | 入参 category 仅接受枚举值；后端校验非法则 400。                                                                      |
| P0-3 | `GET/POST/PUT/DELETE /api/projects/{id}/payment-plans` 或 `/api/payment-plans?projectId=` | 回款计划节点 CRUD；归属项目。                                                                                         |
| P0-3 | `GET /api/dashboard/upcoming-payments` 或 `/api/payment-plans/upcoming`                   | 近期待催款列表：计划日在未来 N 天内且未收齐的节点，含项目、金额、计划日。                                             |
| P0-4 | `GET /api/projects/{id}` 或 `GET /api/projects/{id}/cost-summary`                         | 响应中增加：`materialCostFromFinance`、`materialCostFromStock`、`materialCostTotal`（或等价字段），便于前端分项展示。 |

### P1 相关

| 功能 | 接口/能力                                                                      | 说明                                                                                                |
| ---- | ------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------- |
| P1-1 | `GET/POST/PUT/DELETE /api/suppliers`                                           | 供应商 CRUD。                                                                                       |
| P1-1 | 入库/支出关联供应商                                                            | `StockLog`、`FinanceRecord` 增加 supplierId；创建/更新入库或支出时必填或可选供应商。                |
| P1-1 | `GET /api/suppliers/{id}/balance` 或 `GET /api/reports/supplier-balance`       | 按供应商汇总应付/已付/欠款；供报表页或导出。                                                        |
| P1-2 | `Project` 预算字段                                                             | 新增 totalBudget 或 materialBudget/laborBudget/otherBudget；项目 CRUD 支持读写。                    |
| P1-2 | 支出/出库校验                                                                  | 创建支出或出库时，若关联项目已超预算（达 100% 或锁定状态），拒绝或需管理员权限放行。                |
| P1-2 | `GET /api/projects/{id}` 或预警接口                                            | 返回预算、实际成本、预警状态（如 80% 黄灯、100% 红灯）；或单独 `GET /api/dashboard/budget-alerts`。 |
| P1-3 | `GET/POST/PUT/DELETE /api/change-orders` 或 `/api/projects/{id}/change-orders` | 变更/签证单 CRUD 及审批。                                                                           |
| P1-3 | 审批通过回调                                                                   | 变更单审批通过后，更新 Project.contractAmount 或扩展字段「预估总收」；规则在服务层写死或可配置。    |

### 现有接口需关注的扩展点

- **项目列表/详情**：当前 `ProjectController` 需支持按当前用户过滤（P0-1）及返回成本分项（P0-4）。
- **财务**：`FinanceController` 需支持类别枚举校验（P0-2）；若后续支出关联供应商，需扩展请求体与持久化（P1-1）。
- **仪表盘**：若有统一 Dashboard 接口，需增加待催款（P0-3）、预算预警（P1-2）等数据源。

以上验收标准与接口清单可与「可交付项」「实施顺序」一起作为 Plan Agent 输出 Story、任务及测试检查项的依据。
